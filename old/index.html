记一次面试中的compose

发表于 2019-10-15

Day 180？:
来趣头条了6个多月，技术的成长很多，最近项目遇到瓶颈，另外RN的很多瓶颈越来越多，开发app变得非常掣肘，业务线这边最资深的都想转到H5了。后端混乱的调用关系；能一个需求一下重构2k行+的全量业务需求；每次上线前都能测出好几个版本前问题的让人心惊肉跳的回归测试；还有感觉无奈也不敢相信我们的领导。

最近参加了一些前端的面试，喜马拉雅，爽了的淘集集，CLPS，还有两个小公司，深感前端领域水平的参差不齐。本来coding的根本就是共同的，面试的问题拘泥于水平垂直居中，react-diff算法，手写compose和debounce，浏览器输入网址到显示网页过程，redux的中间件，304错误码，缓存机制，js的闭包等等基础问题，确实是要扎实的功底。

今天去一个Oriente的公司，前端大佬问那个手写一个
compose(a, b, c)(x) => a(b(c(x)))：
const compose = (...args) => (x) => {
    return args.reduce((res, item) => {
        return item(res);
    }, x)
}
搞了半天说思路都错了。。。我愣是没看出来，弄得贼尴尬，回来浏览器一跑。。。这年头，水浅王八多。。

如果我们在一个拥有很多自定义的action简化代理的巨型react项目的调试中希望找到代码中dispatch出action在代码中的位置，我们可以在代理函数中filter指定类型的action并打印new.Error().stack来追溯具体发出action的位置。
另外，今天还碰到一个在rn的项目中，使用了队列来处理安卓机型的物理返回键的对应事件，但在蒙层弹窗期间，有app的screen页面在拦截导航的didFocus事件在蒙层消失之前，插入了队列中，导致蒙层消失后，安卓返回键对应事件的错乱。但是如果针对这种情况做特殊处理，那么会使弹窗代码和navigator部分耦合，目前没想到很好的方法，工作中的大佬领导也没有给出很好的方案。
